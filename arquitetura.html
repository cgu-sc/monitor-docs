⚙ Técnico
Camada 3 — Detalhe

Anti-joins e Consolidação

Cada fonte inclui apenas os registros que não existem nas fontes de maior prioridade, garantindo unicidade sem perda de dados.

Lógica de vw_licitacoes_consolidada_sc:

1. FaseExterna — inclui tudo.
2. Comprasnet — inclui apenas licitações cujo Cod_licitacao não existe no FaseExterna.
3. Siasg DW — inclui apenas licitações que não existem no FaseExterna nem no Comprasnet.
4. PNCP — inclui apenas Dispensas/Inexigibilidades que não existem nas 3 fontes anteriores.
ℹ️
Anti-join de itens a nível de licitação: Se a licitação existe no FaseExterna, todos os seus itens vêm exclusivamente do FaseExterna — mesmo que ele não tenha itens carregados (~2% dos casos).
⚙ Técnico
Camada 2 — Itens · Detalhamento

Riqueza dos Dados de Itens por Fonte

A hierarquia de prioridade para itens é diferente da de licitações porque a disponibilidade de dados varia significativamente entre as fontes.

Prior. Fonte Classif. CATMAT/CATSER Fornecedor Vencedor Valores Adjudicados Qtd Lances Qtd Fornecedores
1ª FaseExterna ✅ Via codigo_item_catalogo → DW ✅ proposta_item + participacao ✅ Hierarquia 4 valores ❌ Não disponível ✅ COUNT(proposta_item)
2ª Siasg DW ✅ D_ITCP_MATERIAL_SERVICO ✅ D_FRND_FORNECEDOR + dimensões ✅ VL_PRECO_UNIT/TOTAL_HOMOLOG ✅ QT_ITCP_LANCES_ITEM ✅ COUNT(F_ITEM_FORNECEDOR)
3ª PNCP ❌ Não disponível ⚠️ Parcial — compraitemresultado ✅ qtdHomologada × valorUnit. ❌ ❌
4ª Comprasnet ⚠️ Parcial — grupo/classe ❌ TBL_PROPOSTAITEM indisponível na CGU ❌ Sem acesso a propostas ❌ ❌
ℹ️
Por que Siasg DW é 2ª para itens, mas 3ª para licitações?
Para licitações, o Comprasnet ocupa a 2ª posição por ter dados estruturais mais recentes (situação, datas) que o Siasg, que parou em abril/2024. Para itens, o Siasg DW é superior: tem classificação completa CATMAT/CATSER e dados do fornecedor vencedor — dados que o db_compras da CGU não oferece por não ter a TBL_PROPOSTAITEM.
🔀 Técnico + Gerencial
Camadas 4 e 5

Stored Procedure + Power BI

A stored procedure resolve dois problemas da consulta de itens de licitações simultaneamente: performance (de +1h para ~5–8 min) e censura de sigilo nos 10 campos monetários.

SQL — Lógica da SP sp_itens_licitacoes_pbi 
-- Passo 1: materializa campos de censura (~3 min, plano isolado)
SELECT Cod_licitacao, Situacao_Gerada, qtd_itens_sigilosos,
       data_primeira_publicacao, data_abertura
INTO #temp_censura FROM vw_licitacoes_consolidada_sc

CREATE INDEX ix_censura ON #temp_censura (Cod_licitacao)

-- Passo 2: JOIN com itens — plano completamente independente (~2 min)
SELECT i.*,
  CASE WHEN c.Situacao_Gerada = 'Não Homologada'
         AND c.qtd_itens_sigilosos > 0
  THEN 0 ELSE i.valor_estimado_calculado END
  -- [+ 9 outros campos monetários censurados da mesma forma]
FROM vw_itens_consolidada_sc i
JOIN #temp_censura c ON i.Cod_licitacao = c.Cod_licitacao
ℹ️
Por que a tabela temporária resolve o timeout? O SQL Server trata views como transparentes e expande o plano completo ao fazer JOIN direto entre as duas views consolidadas. A #temp_censura força planos de execução separados, reduzindo de +1h para ~5–8 min no total.
