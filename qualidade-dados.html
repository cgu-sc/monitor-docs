Problemas Corrigidos

🔴
item.homologado = 'S' como filtro de valor adjudicado
✓ Corrigido v2.0.9 FaseExterna

O campo era exigido como condição para calcular o valor adjudicado. A homologação é registrada a nível de compra, não de item — o campo a nível de item frequentemente não é atualizado.

456.276Licitações homologadas
159.943Com item.homologado='N' mas propostas adjudicadas
35%Percentual com valor adjudicado zerado (incorretamente)
✅
Correção: item.homologado removido das condições de cálculo. Condição suficiente e confiável: proposta.situacao = '6' (Adjudicada). O campo ainda é trazido na view como informação.
🔴
ROW_NUMBER sem desempate por id DESC — seleção aleatória de itens recriados
✓ Corrigido v2.0.9 FaseExterna

O FaseExterna pode recriar itens com novos IDs mantendo o mesmo numero_item e versao = 0. Sem desempate por ID, o banco escolhia um registro arbitrariamente.

Exemplo real: Pregão 90037/2025, UASG 155913 — item 1 tinha 3 IDs com versao=0. O registro mais recente tinha R$ 5.500 negociado; o mais antigo não tinha proposta adjudicada. Sem a correção, o valor aparecia como zero.

✅
Correção: ORDER BY versao DESC, id DESC — o ID maior sempre corresponde ao registro mais recente.
🔴
valor_estimado FaseExterna tratado como global para compras Normais
✓ Corrigido v2.0.8 FaseExterna

A lógica anterior (v2.0–v2.0.7) só multiplicava valor_estimado × quantidade para SRP. Para compras Normais, usava o valor diretamente como total. Verificação empírica em março/2026 confirmou: o campo é sempre unitário no FaseExterna, sem exceção.

✅
Correção: COALESCE(quantidade_solicitada, 1) * COALESCE(valor_estimado, 0) — aplicado sempre, independente do tipo de compra.
🟡
D_ITCP_MATERIAL_SERVICO sem chave única — duplicação de itens no JOIN
✓ Corrigido v2.0.8 Siasg DW

O mesmo código de material/serviço pode existir como Material ('M') e como Serviço ('S') na mesma tabela. JOIN sem desambiguação multiplicava os itens.

✅
Correção: JOIN desambiguado com tipo_item_catalogo. Para o FaseExterna, usa-se TRY_CAST(codigo_item_catalogo AS INT) para evitar falha em valores não-numéricos.
🟡
codGrupo com valores sentinela (0, -1, -2, -3) tratados como grupo válido
✓ Corrigido v2.0.8 Comprasnet

Valores sentinela do campo codGrupo eram exibidos no painel como números de grupo válidos.

✅
Correção: CASE WHEN codGrupo > 0 THEN codGrupo ELSE NULL END
🟡
DT_PUBLICACAO_DOU com 88% de valores inválidos (1900-01-01) no Siasg
✓ Corrigido v2.0.3 Siasg DW
86%Dispensas com data inválida
90%Inexigibilidades com data inválida
0%DT_CMPR_RESULTADO_COMPRA inválidos
✅
Correção: Fallback para DT_CMPR_RESULTADO_COMPRA quando DT_PUBLICACAO_DOU <= '1901-01-01'. Este campo está 100% preenchido para todas as modalidades.
🟡

O anti-join da consolidada de itens opera a nível de licitação — se ela existe no FaseExterna, os itens do Comprasnet são excluídos mesmo que o FaseExterna não tenha itens. Investigação mostrou que os itens do Comprasnet nesses casos também estão sem valores (ipgValorRef = NULL).

Sem impacto prático PNCP

Pode conter valores como "060301/2024", "DL 1" e "10 | Processo 212024". Sem impacto — o PNCP só é usado para registros inexistentes nas outras fontes, sem necessidade de JOIN por esse campo.

🟡
Valores adjudicados do Comprasnet indisponíveis após abril/2024
Monitorar Siasg DW

O Siasg DW é a única fonte de valores adjudicados para licitações do Comprasnet disponível na CGU. Como parou de ser atualizado em abril/2024, licitações do Comprasnet posteriores a essa data não têm valor adjudicado. Para licitações de 2024+, o FaseExterna é a fonte principal e tem os valores completos.
